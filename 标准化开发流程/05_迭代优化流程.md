### 2.2 快速沟通技巧

**如果客户说得不清楚**：

| 客户说法 | 追问方式 |
|---------|---------|
| "这里不太对" | "能具体说一下哪里不对吗？是样式还是功能？" |
| "感觉怪怪的" | "是颜色、大小、位置、还是交互让您不舒服？" |
| "能不能改一下" | "您希望改成什么样？有参考的例子吗？" |
| "再加个功能" | "这个功能主要解决什么问题？使用场景是什么？" |

**让客户更好表达**：
- 发对比图：现在是A，改成B可以吗？
- 给选项：方案1是...，方案2是...，您选哪个？
- 问场景：什么情况下需要用这个功能？

## 三、问题分类

### 3.1 分类标准

| 类型 | 说明 | 示例 |
|------|------|------|
| **Bug** | 功能不正常 | 点击按钮没反应、页面崩溃 |
| **UI调整** | 视觉效果调整 | 颜色、字体、间距、布局 |
| **交互优化** | 操作体验优化 | 加载慢、流程繁琐、反馈不明确 |
| **功能变更** | 需求变更 | 增加字段、修改逻辑、删除功能 |
| **新功能** | 新增需求 | 原来没有的功能 |

### 3.2 问题记录表

```markdown
| 编号 | 类型 | 描述 | 页面 | 优先级 | 状态 | 备注 |
|------|------|------|------|--------|------|------|
| 001 | Bug | 首页列表不显示 | /pages/home | P0 | 待修复 | |
| 002 | UI | 按钮颜色太暗 | 全局 | P2 | 待修复 | |
| 003 | 新功能 | 增加搜索功能 | /pages/home | P1 | 评估中 | 需讨论范围 |
```

## 四、优先级排序

### 4.1 优先级定义

| 等级 | 定义 | 响应时间 |
|------|------|---------|
| **P0 紧急** | 阻塞使用、核心功能不可用 | 立即处理 |
| **P1 高** | 影响体验但可以绕过 | 本轮迭代 |
| **P2 中** | 改了更好 | 下轮迭代 |
| **P3 低** | 锦上添花 | 放入计划 |

### 4.2 排序原则

1. **Bug优先于功能**：先修复bug，再加新功能
2. **核心路径优先**：影响主流程的优先
3. **影响面优先**：影响多个页面的优先
4. **客户强调的优先**：客户反复强调的优先

### 4.3 与客户确认优先级

```markdown
## 本轮计划修改（请确认）

### 必须做（P0/P1）
1. [x] 修复首页列表不显示的bug
2. [x] 调整登录按钮点击无反应的问题
3. [x] 增加下拉刷新功能

### 尽量做（P2）
4. [ ] 调整按钮颜色
5. [ ] 优化加载速度

### 暂不做（P3/需评估）
6. [ ] 搜索功能（需要进一步讨论需求）
7. [ ] 分享功能（考虑下一版本）

**请确认以上安排是否合适？如有调整请说明。**
```

## 八、迭代管理技巧

### 8.1 控制每轮修改量

| 轮次 | 建议内容 | 预计耗时 |
|------|---------|---------|
| 第1轮 | 核心流程验证 | 1-2天 |
| 第2轮 | UI风格确定 | 1天 |
| 第3轮 | 细节调整 | 半天 |
| 后续 | 新功能迭代 | 视功能而定 |

### 8.2 管理需求蔓延

当客户不断增加需求时：

1. **先记录**：所有需求都记录下来，不要拒绝
2. **再分类**：和客户一起确定优先级
3. **说明影响**：新需求会影响进度/成本
4. **分批做**：核心功能先上线，其他后续迭代

**话术示例**：
> "这个功能确实很好，我先记录下来。不过为了确保核心功能尽快上线，建议我们先把报名流程做完，这个搜索功能放在1.1版本做，您看可以吗？"

### 8.3 处理反复修改

当同一个地方反复修改时：

1. **提供选项**：一次给2-3个方案让客户选
2. **看参考**：让客户找喜欢的案例
3. **定规范**：确定后记录到设计规范，后续统一
4. **适当引导**：从专业角度给建议

## 九、版本管理

### 9.1 版本命名规则

```
v主版本.功能版本.修复版本

例如：
v0.1.0 - 第一个Demo
v0.2.0 - 第二轮迭代
v0.2.1 - 修复小bug
```

```bash
# 每次大的迭代前备份

git add .
```

## 十、常见问题处理

### 10.1 客户响应慢

**策略**：
1. 设定截止日期："请在X日前反馈，以便安排开发"
2. 提供默认选项："如无反馈，将按当前方案继续"
3. 分批确认：不要一次发太多内容

### 10.2 需求来回变

**策略**：
1. 书面确认：重要决定通过文字确认
2. 说明成本："这个修改需要额外X时间"
3. 限制次数："这个位置已修改2次，建议确定后再改"

### 10.3 沟通不畅

**策略**：
1. 换方式：文字不清楚就语音/视频
2. 用可视化：截图、对比图、流程图
3. 当面沟通：关键决策安排会议
